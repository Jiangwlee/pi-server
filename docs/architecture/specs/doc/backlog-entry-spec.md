# Backlog Entry 写作规范

本规范定义 `docs/backlog.md` 单条条目的必备字段、登记阈值、命名与交叉引用规则。每条 backlog 按此模板填写。

---

## 0. 核心原则

**backlog.md 只记录尚未解决的问题。**

- **登记**：发现符合阈值的缺陷/债务时，立即新增条目。
- **删除**：问题修复后，直接从 `backlog.md` 中删除该条目——不保留、不标注状态。
- **溯源**：关闭时在 commit message 中注明 `closes backlog: <条目标题>`，git 历史即为档案。

> 禁止以"已关闭"状态标注保留条目。保留已关闭条目会稀释可读性，让 backlog 丧失行动信号。

---

## 1. 适用范围

- 适用：`docs/backlog.md` 新增 / 修改 / 关闭条目
- 不适用：单条代码改动的 TODO 注释（TODO 必须带时间边界和归属，属代码范畴，见 AGENTS.md 开发模式第 7 条）

---

## 2. 必备字段（6 字段模板）

每条 backlog 按以下顺序包含：

### 2.1 标题（H2 heading）

一句话描述问题。**禁止**以 "TODO:" / "需要" / "要不要" 开头。

好例：
- `Hermes agents 不能作为 Team 成员`
- `Team DELETE session 不 cancel 进行中的 run + 不清 registry 内存 entry`

坏例：
- `修一下 chat 路由`
- `MessageAdapter 好像有点问题`

### 2.2 意图

**一段话**，说明"理想状态"是什么——问题解决后系统该呈现的样子。**不要**描述现状。

好例：
> Team 成员应当可以来自任一 Agent Provider（OpenClaw / Hermes / 未来 pi-server），用户能在 Team 编辑器中无差异地选择。

### 2.3 现状差距

**bullet list**，列出**可验证**的事实差距。每条：

- 指向**具体文件 + 行号**（若适用）
- 引用代码片段或符号名
- 说明具体偏离点

好例：
- `` `/api/chat/send`（`src/app/api/chat/send/route.ts`）只实现 `provider === 'openclaw'`（第 98 行起）与 `provider === 'hermes'`（第 217 行起）两个分支 ``

坏例：
- `Chat 路由好像不完整`

### 2.4 拟议方案

**numbered list** 或 **bullet list**，列出候选修复路径。多方案时**标明推荐**。每条至少含：

- 动作描述
- 影响面预估
- 与其他 backlog 条目的依赖关系（若有）

允许"待设计"或"待决策"，但需指明缺失什么信息。

### 2.5 影响面

**bullet list**，列出**修复会触及的文档和代码**：

- 影响哪些 module 文档
- 影响 `architecture.md` 的哪一节
- 影响 `configuration.md` 的哪一节
- 影响哪些测试
- 对用户可观察到的行为变化

### 2.6 相关文件

**bullet list**，列出与问题直接相关的源文件（含**行号**若精确）。每条格式：

```
- `src/xxx/yyy.ts`（简述其在此问题中的角色）
```

### 2.7 来源

**一行**，说明条目被识别的时机与上下文。格式：

```
**来源**：`<触发工作>` 起草/审查过程中（YYYY-MM-DD）<触发动作> 时发现。
```

好例：

```
**来源**：`modules/chat-pipeline.md` 起草过程中（2026-04-22）审查 `src/app/api/chat/*` 路由时发现。
```

## 3. 登记阈值

见 [AGENTS.md §持续审查与债务登记 · 登记阈值](../../../../AGENTS.md)。简述：

| 该进 backlog | 不该进 backlog |
|---|---|
| 违反 AGENTS.md 已声明原则 | 个人不同意某种风格 |
| 接口声明但实现不一致 | 命名可以更好但语义清晰 |
| 死代码 / 死端点 | 未使用的 import |
| feature 缺口 | 想要更花哨的 UI |
| 模块边界渗漏 | 文件太长 / 函数太长 |
| 同名对象双版本 / 术语不对齐 | 个人审美偏好 |

**关键判断标准**：问题是否**系统级 / 影响多个文件 / 破坏某个原则 / 形成反模式**。若是则登，否则不登。

---

## 4. 禁止

- ❌ **禁止**无文件路径的抽象描述——每一条"现状差距"应可被 grep 或 IDE 跳转验证
- ❌ **禁止**"个人认为"/"可能"/"也许"的推测语——要么是事实，要么附代码证据
- ❌ **禁止**条目之间内容重复——同一问题只登一次，相关条目互相引用
- ❌ **禁止**用 backlog 记录"未来想要的 feature"——feature 请求应放 roadmap，不放 backlog
- ❌ **禁止**登记拼写修正 / 格式化 / lint 建议——这些走 linter 或 PR 评审

---

## 5. 维护约定

### 5.1 新增条目时

- 立即在正确位置插入（保持时间倒序或按议题分组）
- 同步更新相关 `modules/<name>.md` §9（若是 module 独有缺口）
- 在 AGENTS.md / architecture.md / module 文档中**反向链接**时使用形式 `docs/backlog.md 第 N 项`

### 5.2 关闭条目时

- **直接删除**该条目——不保留、不标注状态
- 在 commit message 中注明 `closes backlog: <条目标题>`，git 历史即为溯源档案
- 从 module 文档的 §9 移除该条目指针

### 5.3 条目内容变更时

- 只允许补充新事实或新方案
- 历史内容保留（可在末尾加 `**补充（YYYY-MM-DD）**：...` 段）

---

## 6. 模板骨架

新增 backlog 条目时复制以下骨架填写：

```markdown
## <一句话问题描述>

**意图**：<一段话，理想状态>

**现状差距**：

- <可验证事实 1，含文件路径/行号>
- <可验证事实 2>

**拟议方案**：

1. **<推荐方案名>**：<动作描述 + 影响面估算>
2. **<备选方案>**：...

**影响面**：

- 影响 <架构文档/module/配置/测试>
- 用户可观察到的行为变化：<描述>

**相关文件**：

- `src/xxx/yyy.ts`（<角色>）
- `src/xxx/zzz.ts`（<角色>）

**来源**：`<触发工作>` 过程中（YYYY-MM-DD）<触发动作>时发现。
```
